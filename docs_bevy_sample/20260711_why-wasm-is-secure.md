# WASM はなぜ「セキュア」なのか（守るのはホスト、砂場の中身は別責任）

日付: 2026-07-11

WASM が「セキュア」と言われる意味を、このリポジトリの Rust(WASM)→JS ブリッジの
実例（`web_sys`/`js_sys` 経由で `window.__BREAKOUT_CONFIG__` を読む仕組み）と
結びつけて整理した概念メモ。初期化パラメータ注入の具体的な実装は
[[20260711_react-to-bevy-init-params]] を参照。

---

## 1. 核心の一文

WASM が「セキュア」なのは、**動かしているコードからホスト（ブラウザ / OS）を守る**
という一点に尽きる。JS と同じサンドボックス内で動くよう設計されているため、
untrusted なコードを安全にブラウザで走らせられる、という Web の前提に合わせた設計。

## 2. セキュアである理由（5点）

1. **サンドボックス**
   WASM モジュールには OS 機能への直接アクセス手段が無い。syscall を自分で呼べない
   （任意の OS API を叩けるネイティブ実行ファイルとは対照的）。

2. **linear memory（箱に閉じたメモリ）**
   使えるメモリは 1 本の連続配列だけ。ポインタは「配列の何番目か」という数値インデックス
   であって、ホストの実アドレスではない。範囲外アクセスは即トラップ。
   → 中の C/Rust にバッファオーバーフローのバグがあっても、壊せるのは自分の箱の中だけで、
   ホストのメモリには届かない。

3. **制御フロー整合性（CFI）**
   コールスタック（関数の戻り先）は linear memory とは別領域にあり、プログラムから触れない。
   ジャンプ先も関数テーブルの正当なエントリに限定される。
   → 戻りアドレス上書きによる任意コード実行という古典的攻撃が効きにくい。

4. **capability ベース**
   モジュールはホストが明示的に import として渡した関数しか呼べない。DOM も fetch も、
   JS が手渡さない限り触れない。
   → **このプロジェクトの実例**: Rust(WASM) は `window.__BREAKOUT_CONFIG__` を勝手に
   読めない。`web_sys` / `js_sys` という「ホストが渡した窓口」を通してのみ `window` に
   アクセスし、React が注入した背景 URL やブロック定義を受け取る。窓口が渡されなければ
   その先には一切到達できない（詳細は [[20260711_react-to-bevy-init-params]]）。

5. **実行前の検証（validation）**
   型・命令列の正当性を実行前に検証する。壊れた／不正な命令列はロード段階で弾かれ、
   実行に到達しない。

## 3. 誤解しやすい点（重要）

- **「WASM がセキュア」＝「WASM で書いたプログラムにバグ / 脆弱性が無い」ではない。**
- 守られるのはホスト側（ブラウザ・OS・他サイト）。箱の中は別問題で、メモリ安全でない
  言語で書けば、**自分の linear memory 内**でのオーバーフローや情報漏洩は普通に起こり得る。
- JS 経由で渡した capability は、その範囲内で悪用され得る。
  → 渡す capability は最小限に絞る（本プロジェクトでも window から読むのは設定値に限定）。

## 4. まとめ

WASM は「OS の鍵を渡さず、閉じた砂場の中で、範囲外に触れたら即停止するよう設計された
実行環境」だからセキュア。守っているのは **“外”（ホスト）** であって、砂場の中のコード品質は
別途開発者の責任、というモデル。このプロジェクトの `web_sys`/`js_sys` 越しの window 読み取りは、
capability ベースという設計思想がそのまま形になった具体例と言える。
